- Using linked lists for fields and methods and arguments because they are not encounterd in order,
  so you don't know the length of them until you're done parsing.

- Explicit naming of cases so you can name types from other peoples' code.
- No duplicate fields or methods -> no conflicts when implementing traits or embedding structs
- Unique fields and methods means that in order for two structs to be compatible one must embed the
  other or name the other as a case. This also means traits can't have fields because to get them
  you'd have to embed the trait explicitly which would prevent new types from implementing the
  trait.
- Struct vs. trait because is a type with no fields and all concrete cases abstract or concrete?
- Struct vs. trait is necessary because a type with private fields and public methods looks like an
  interface that anyone can implement but actually is not.
- Fields are by nature inheritance-based and nominal, methods are by nature implementation-based and
  structural.
- Methods can be variant on more than one parameter.
- Methods are defined outside of trait bodies in the module namespace because this makes them
  identical to functions and reinforces that they don't belong to any type. This also makes it easy
  to refactor a function into a method and makes it so if you change a struct to a trait or vice
  versa you don't need to refactor any code that uses it and calls functions on it.
- This plus properties on traits makes it completely identical to use both.
- If a function from another module doesn't have the arguments in the right order to implement a new
  trait you wrote, you can just write a function with the arguments in the right order that calls
  the old function. This way you never have to decide what type a function "belongs" to in an OO
  sense.
- Embedding a type with cases in another type only embeds the value cases. If you want to mirror the
  names of the type cases you must define them on your own.
- Embedding a trait in another trait exports the methods that are variant over that trait.
- You can't define a new method for a trait outside of the module it's defined in.
- If you want a struct to contain two other structs that have overlapping fields, you have to use
  composition, you can't embed both.
- Forcing fields and methods to be unique and making methods and variants in the namespace of the
  module forces you to make flat apis.
- Value cases are defined inside of the "cases {}" block, type cases are defined outside. This makes
  it trivial to switch a type between being opened and closed.

}

	foo.data = allocate(a, &1)

	free(allocator, foo.data)
}
